## Why Binary?

Explain the difference between lossless and lossy compression, and discuss when each would be appropriate for image storage.

> [!success]- Answer
> **Answer:**
> 
> **Lossless Compression:**
> 
> - Preserves all original data perfectly
> - Original file can be reconstructed exactly from compressed version
> - Achieves compression by eliminating redundancy (e.g., run-length encoding, dictionary methods)
> - Typical compression ratios: 2:1 to 5:1
> - Examples: PNG, GIF, ZIP, FLAC (audio)
> - No quality degradation
> 
> **Lossy Compression:**
> 
> - Permanently removes data that is less important or redundant
> - Original file cannot be perfectly reconstructed
> - Uses perceptual coding (removes data human senses are less likely to notice)
> - Typical compression ratios: 10:1 to 50:1 or higher
> - Examples: JPEG, MP3, MP4, H.264
> - Some quality degradation, but often imperceptible at reasonable compression levels
> 
> **When to Use Lossless:**
> 
> 1. **Professional Photography/Editing**: When images need to be edited, cropped, or printed at large sizes without quality loss
> 2. **Medical/Scientific Imaging**: Where data accuracy is critical (X-rays, satellite imagery)
> 3. **Archival Storage**: When preserving exact quality for future use
> 4. **Graphics with Text/Sharp Edges**: PNG for screenshots, logos, diagrams
> 5. **Multiple Edits**: Lossless prevents generation loss from repeated compression
> 
> **When to Use Lossy:**
> 
> 1. **Web/Online Display**: JPEG for photos on websites where file size matters for loading speed
> 2. **Consumer Photography**: When storage space is limited and minor quality loss is acceptable
> 3. **Social Media**: Where images are viewed on small screens and quality loss is less noticeable
> 4. **Streaming/Sharing**: When bandwidth or storage is a constraint
> 5. **Thumbnails/Previews**: Lower quality is acceptable for preview images
> 
> **Mark Scheme:**
> - Clear explanation of lossless compression (preserves all data) [1 mark]
> - Clear explanation of lossy compression (removes data) [1 mark]

> **(c) Additional Technique:**
> 
> **Asset Streaming/On-Demand Loading:**
> 
> Instead of including all assets in the initial download, implement an asset streaming system:
> 
> - **Initial Download**: Include only essential assets needed to launch the game (main menu, first level basics)
> - **Progressive Download**: Download additional assets (backgrounds, music, later levels) as needed during gameplay
> - **Caching**: Store downloaded assets on device for future use
> - **Level-Based Loading**: Load assets only when entering specific levels or areas
> 
> **Benefits:**
> - Reduces initial download size significantly (could be <50 MB initial download)
> - Faster initial installation and startup
> - Better user experience (game playable sooner)
> - Allows higher quality assets to be downloaded when needed
> - Reduces RAM usage (only loaded assets consume memory)
> 
> **Implementation Considerations:**
> - Requires network connectivity for asset downloads
> - Need progress indicators for downloads
> - Must handle offline scenarios gracefully
> - Requires efficient asset management system
> 
> **Alternative Techniques:**
> - **Texture Atlasing**: Combine multiple small images (sprites, UI) into single texture sheets to reduce overhead
> - **Mipmaps**: Pre-generate smaller versions of images for different display resolutions
> - **Audio Streaming**: Stream music during gameplay rather than pre-loading all tracks
