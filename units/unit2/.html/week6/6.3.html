Different CPU Scheduling Approaches and Their Impact on System Performance

CPU scheduling is a crucial function of an operating system that determines how processes are assigned CPU time to optimize system performance, responsiveness, and resource utilization. Since multiple processes often compete for CPU time, the OS employs scheduling algorithms to allocate time efficiently and fairly. Different scheduling approaches impact system performance in terms of speed, efficiency, process fairness, and user experience. The choice of scheduling algorithm depends on factors such as process priority, execution time, and system workload.

Common CPU scheduling approaches include First-Come-First-Served (FCFS), Round Robin (RR), Multilevel Queue Scheduling, and Priority Scheduling. Each has distinct advantages and disadvantages, making them suitable for different types of computing environments such as real-time systems, interactive applications, and batch processing systems.

How Process Scheduling Optimizes Execution by Allocating CPU Time Efficiently

In a multitasking computing environment, multiple processes compete for CPU time. Since the CPU can only execute one process at a time per core, the operating system (OS) employs process scheduling to determine the order and duration for which each process receives CPU access. Process scheduling optimizes execution by ensuring fair allocation, efficient resource utilization, and minimal waiting time, leading to an improved user experience, system responsiveness, and overall efficiency.

Without effective process scheduling, certain tasks may experience long delays, leading to system slowdowns, high latency, or even unresponsive applications. By managing CPU allocation using scheduling algorithms, modern operating systems maintain seamless multitasking, prevent resource starvation, and prioritize time-sensitive processes.

Different CPU Scheduling Approaches

1.  First-Come-First-Served (FCFS) Scheduling

FCFS scheduling follows a simple rule: the process that arrives first gets executed first. It operates like a queue, where the first process to enter is the first to use the CPU. However, this method can lead to inefficiencies due to the convoy effect, where short tasks get stuck behind long ones.

Algorithm: First-Come-First-Serve (FCFS) Scheduling using Stack

Assumptions:

  • Each process has:
    • Process ID (PID)
    • Arrival Time
    • Burst Time
  • Input is given in a stack named InputStack, with the last process on top.
  • To simulate FCFS using stack (LIFO), we reverse the stack into ReverseStack.

Step 1: Initialization

  • InputStack ← stack of processes (each with PID, ArrivalTime, BurstTime)
  • ReverseStack ← empty stack
  • CurrentTime ← 0
  • Output ← empty list

Step 2: Reverse the Input Stack

While InputStack is not empty:
 process ← InputStack.pop()
 ReverseStack.push(process)

Step 3: Simulate FCFS Execution

While ReverseStack is not empty:
 process ← ReverseStack.pop()
 If CurrentTime < process.ArrivalTime:
  CurrentTime ← process.ArrivalTime
 StartTime ← CurrentTime
 CompletionTime ← StartTime + process.BurstTime
 TurnaroundTime ← CompletionTime - process.ArrivalTime
 WaitingTime ← TurnaroundTime - process.BurstTime
 Add (process.PID, StartTime, CompletionTime, TurnaroundTime, WaitingTime) to Output
 CurrentTime ← CompletionTime

Step 4: Display Output

For each record in Output:
 Print PID, StartTime, CompletionTime, TurnaroundTime, WaitingTime

Real-World Example: Bank Teller Service

In a bank, customers are served in the order they arrive. If one customer has a complex request, others must wait longer, leading to inefficiencies. Similarly, in FCFS scheduling, long processes delay shorter processes, increasing the average wait time.

Impact on Performance

  • Simple and easy to implement but inefficient for interactive systems.
  • Longer average waiting time, especially if a long process arrives first.
  • No prioritization, making it unsuitable for time-sensitive applications.

2.  Round Robin (RR) Scheduling

Round Robin scheduling allocates equal CPU time (time quantum) to each process in a cyclic order. If a process does not finish within the given time slice, it is moved to the back of the queue, and the next process gets CPU access.

Algorithm: Round Robin Scheduling

Input:

  • List of processes with: Process ID, Arrival Time, Burst Time
  • Time Quantum (Q)

Steps:

  1. Initialize a queue Q
  2. Set current time T = 0
  3. Create an empty list called Completed
  4. Sort the processes by Arrival Time
  5. Set index = 0
  6. While index is less than the number of processes and process[index].arrival_time <= T:
    • Add process[index] to queue Q
    • Increment index
  1. While the queue Q is not empty:
    • Remove the first process P from the queue
    • If P.remaining_time > Q:
      • Run P for Q time units
      • Increment T by Q
      • Subtract Q from P.remaining_time
    • Else:
      • Run P for its remaining time
      • Increment T by P.remaining_time
      • Set P.remaining_time = 0
      • Set P.completion_time = T
      • Add P to Completed list
    • Add any newly arrived processes (arrival_time <= T) to queue Q
    • If P.remaining_time > 0, add P back to the queue
  1. After all processes are complete, for each process:
    • Turnaround Time = Completion Time - Arrival Time
    • Waiting Time = Turnaround Time - Burst Time
    • Display the process info and metrics

Real-World Example: Time-Sharing in Multiplayer Online Games

In online multiplayer games, the server allocates a fixed time slice to each player’s actions to ensure smooth gameplay. No single player can dominate the server, allowing fair access to game resources.

Impact on Performance

  • Ensures fairness as each process gets CPU time in equal intervals.
  • Ideal for interactive systems such as time-sharing environments and real-time applications.
  • May cause overhead due to frequent context switching, reducing overall efficiency.

3.  Multilevel Queue Scheduling

Multilevel queue scheduling categorizes processes into different queues based on their priority, type, or resource needs. Each queue follows its own scheduling algorithm, and the OS determines how to allocate CPU time between queues.

Algorithm: Multilevel Queue Scheduling

Assumptions:

  • There are multiple queues for different types of processes (e.g., system processes, interactive processes, batch processes).
  • Each queue has its own scheduling algorithm (e.g., Round Robin for one, FCFS for another).
  • A fixed priority is assigned to each queue; no process moves between queues.

 

Steps:

  1. Define the number of queues, e.g., 3 queues:
    • Queue 1: System Processes (Highest Priority)
    • Queue 2: Interactive Processes
    • Queue 3: Background/Batch Processes (Lowest Priority)
  1. Assign each incoming process to a specific queue based on its type or priority.
  2. Use a scheduling algorithm within each queue:
    • Queue 1: Round Robin with quantum = 4
    • Queue 2: FCFS (First-Come, First-Served)
    • Queue 3: Shortest Job First (SJF), or another policy
  1. At each scheduling cycle:
    • Start checking from the highest priority queue (Queue 1).
    • If Queue 1 has processes, schedule based on its algorithm.
    • Only if Queue 1 is empty, move to Queue 2.
    • If Queue 2 is also empty, schedule from Queue 3.
  1. Repeat the cycle until all queues are empty and all processes are completed.

Real-World Example: Air Traffic Control System

In air traffic control, flights are categorized into emergency landings, commercial flights, and cargo flights. Emergency landings always get top priority, while commercial and cargo flights follow different scheduling rules.

Impact on Performance

  • Separates high-priority tasks from low-priority tasks, improving responsiveness.
  • Best suited for multitasking environments where different types of tasks need CPU access.
  • Can lead to starvation if lower-priority processes are never given CPU time.
Key Points:
  • No starvation protection unless aging is introduced.
  • Processes are permanently assigned to one queue.
  • Different policies allow flexibility based on process behavior.

4. Priority Scheduling

Priority scheduling assigns a priority level to each process, and the CPU executes the highest priority process first. If multiple processes have the same priority, they are scheduled using FCFS.

Algorithm: Priority Scheduling

Assumptions:

  • Each process is assigned a priority number.
  • Lower numbers can represent higher priority (or vice versa, depending on the convention).
  • In case of equal priority, a tie-breaker like FCFS (First-Come, First-Served) is used.
  • Preemptive or non-preemptive versions may be used.

 

Steps (Non-Preemptive Priority Scheduling):

  1. Start with a list of processes, each with:

    • Process ID

    • Arrival Time

    • Burst Time

    • Priority

  1. Sort the list of processes by Arrival Time.

  2. From the list of arrived processes (at current time):

    • Select the process with the highest priority (lowest priority number if using ascending order).

    • If multiple processes have the same priority, use First-Come-First-Served to break the tie.

  1. Execute the selected process completely (non-preemptive).

  2. Update the current time after the process finishes.

  3. Repeat steps 3–5 until all processes are completed.

 

Steps (Preemptive Priority Scheduling):

  1. At every unit of time, check for newly arrived processes.

  2. From the ready queue, select the process with the highest priority.

  3. If a newly arrived process has higher priority than the currently running one:

    • Preempt the current process.

    • Start executing the higher-priority process.

  1. Continue until all processes are completed.

Real-World Example: Hospital Emergency Room

In a hospital emergency room, patients with life-threatening conditions are treated first, while those with minor injuries wait longer. Similarly, in priority scheduling, higher-priority processes are executed before lower-priority ones.

Impact on Performance

  • Ensures critical tasks are executed first, improving response time for important processes.
  • Can lead to starvation, where low-priority tasks never get executed.
  • Aging techniques can be used to gradually increase the priority of long-waiting processes to prevent starvation.
Key Points:
  • Can lead to starvation if low-priority processes never get CPU time.
  • Aging can be implemented to gradually increase the priority of waiting processes.
  • Preemptive version is more responsive but more complex to manage.

Key Points to Remember
  • CPU scheduling optimizes performance by deciding how processes share CPU time.
  • FCFS is simple but inefficient for real-time systems due to long wait times.
  • Round Robin is best for interactive systems, ensuring fair access but increasing context-switching overhead.
  • Multilevel Queue Scheduling classifies processes into different queues, optimizing multitasking environments.
  • Priority Scheduling prioritizes critical tasks but requires starvation prevention techniques like aging.
  • Different scheduling algorithms are chosen based on system requirements and workload types.
